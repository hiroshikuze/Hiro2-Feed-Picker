# Revision Log

セッション開始時にこのファイルを読み、過去のミスパターンを把握してから作業を始める。

ミス発生時は以下の形式で追記する：

```
## YYYY-MM-DD
- **ミスの種類**: （例: typo修正漏れ、設計判断の見落とし）
- **何が起きたか**:
- **なぜ起きたか**:
- **次回の対策**:
```

---

## 2026-03-30
- **ミスの種類**: typo の見落とし
- **何が起きたか**: `sendOwnerNotigication` という関数名のtypoを、CLAUDE.md作成時に「typoのまま維持すること」と誤ってドキュメント化した
- **なぜ起きたか**: 既存コードのtypoを「意図的な命名」と誤解した
- **次回の対策**: 関数名・変数名に違和感があれば既存コードをGrep確認し、typoか意図的かを判断してから記録する

## 2026-06-13
- **ミスの種類**: モデル名の誤読・誤った前提に基づく実装
- **何が起きたか**: 請求書のSKU名「Generate content output token count gemini 3.5 flash text」を、`main.js` が使用する `gemini-2.5-flash` の表記揚れだと推測し、検証せずに `thinkingConfig: { thinkingBudget: 0 }` を実装・デプロイした。実際には「Gemini 3.5 Flash」は別の実在モデルで、`main.js` では一切呼び出されておらず、本来のコスト主因とは無関係だった。
- **なぜ起きたか**: SKU名の「3.5」と「2.5」という明確な差異を「Googleの請求システム特有の表記揚れ」と決めつけ、AI Studioの使用状況など別の証拠で実在モデルかどうかを確認しなかった。
- **次回の対策**: 請求書・ログに登場するモデル名・SKU名は、コード内の記述と文字列として完全一致するか確認する。一致しない場合は「表記揚れ」と決めつけず、まず実在の別モデルである可能性をAI Studio使用状況ページ等で調査する。
